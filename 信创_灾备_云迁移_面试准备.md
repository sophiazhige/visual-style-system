### 1.4 信创云

信创云是信创体系在云计算领域的延伸，要求云平台底层全栈国产化。

**信创云的核心要求：**
- 底层硬件采用国产CPU（鲲鹏/飞腾/海光）
- 虚拟化/容器平台基于国产OS（麒麟/统信）
- 云平台管理软件自主可控
- 数据库、中间件等PaaS层全部国产化

**主要信创云平台：**
- **天翼云**：中国电信，央企背景，政务云市场份额领先
- **华为云Stack**：全栈信创，鲲鹏生态
- **移动云**：中国移动
- **联通云**：中国联通
- 阿里云飞天信创版、腾讯云信创版

**天翼云在信创领域的优势：**
- 央企身份，天然满足政务/国企采购资质要求
- 全国31省资源池布局，属地化服务能力强
- 云网融合——电信骨干网+云平台深度绑定
- 已通过多项信创认证，与主流信创厂商完成适配

## 二、灾备云（Disaster Recovery Cloud）

### 2.1 RPO 和 RTO 定义

| 指标 | 全称 | 定义 | 通俗解释 |
|------|------|------|---------|
| **RPO** | Recovery Point Objective | 灾难发生后，数据必须恢复到的时间点 | **能丢多少数据**（往前看） |
| **RTO** | Recovery Time Objective | 灾难发生后，系统从停顿到恢复的最长时间 | **能停多长时间**（往后看） |

### 2.2 不同等级的RPO/RTO参考值

| 等级 | RPO | RTO | 典型场景 | 成本 |
|------|-----|-----|---------|------|
| **Tier 1（最高）** | ≈ 0（秒级） | < 1分钟 | 金融核心交易系统、证券 | 极高 |
| **Tier 2** | < 15分钟 | < 30分钟 | 银行非核心系统、运营商计费 | 高 |
| **Tier 3** | < 1小时 | < 4小时 | 政务系统、ERP | 中等 |
| **Tier 4** | < 24小时 | < 24小时 | 开发测试环境、非核心OA | 低 |

### 2.3 三大灾备架构

#### （1）同城双活（Active-Active，同城两个数据中心）

```
┌──────────┐    数据同步复制     ┌──────────┐
│ 数据中心A │ ◄══════════════► │ 数据中心B │
│ （生产）  │   距离：30-100km   │ （生产）  │
│ 承载50%   │   延迟 < 2ms      │ 承载50%   │
└──────────┘                   └──────────┘
```

- **特点**：两个中心同时对外提供服务，实时同步
- **RPO ≈ 0，RTO < 30秒**
- **优势**：资源利用率高（不浪费备用中心），切换速度快
- **劣势**：无法应对区域性灾难（地震、大范围断电）
- **适用**：金融、运营商等高可用场景

#### （2）异地灾备（Active-Standby，远距离备份中心）

```
┌──────────┐    异步数据复制     ┌──────────┐
│ 生产中心  │ ═══════════════► │ 灾备中心  │
│ （主用）  │   距离：> 200km   │ （备用）  │
│ 承载100%  │                  │ 待命状态  │
└──────────┘                   └──────────┘
```

- **特点**：灾备中心平时不对外服务，异步复制数据
- **RPO：分钟到小时级，RTO：小时级**
- **优势**：可应对区域性灾难
- **劣势**：灾备中心资源闲置，切换时间长
- **适用**：政务、一般企业

#### （3）两地三中心（最完整方案）

```
              同城（< 100km）
┌──────────┐  同步复制  ┌──────────┐
│ 生产中心  │ ◄═══════► │ 同城灾备  │
│ （主用）  │           │ （双活）  │
└─────┬────┘           └──────────┘
      │ 异步复制
      │ 距离 > 200km
      ▼
┌──────────┐
│ 异地灾备  │
│ （冷/温备）│
└──────────┘
```

- **特点**：同城双活 + 异地灾备的结合
- **同城之间**：同步复制，RPO ≈ 0，RTO < 分钟级
- **异地之间**：异步复制，RPO 分钟级，RTO 小时级
- **优势**：既保证高可用，又能应对区域性灾难
- **适用**：金融核心系统、大型央企（监管硬性要求）
- **中国人民银行要求**：核心系统必须"两地三中心"

### 2.4 天翼云在灾备领域的优势

| 优势维度 | 具体说明 |
|----------|---------|
| **骨干网络** | 中国电信拥有国内最大的骨干网络（ChinaNet + CN2），跨地域数据同步延迟低、带宽有保障 |
| **资源池分布** | 全国"2+4+31+X"资源池布局——2个核心节点+4个区域中心+31省节点+X个边缘节点，天然适合构建两地三中心 |
| **云网融合** | 电信网络+云平台一体化，减少中间跳转环节，灾备链路可靠性高 |
| **属地化服务** | 各省均有本地团队，灾备演练、故障响应速度快 |
| **合规优势** | 央企身份+等保/密评资质齐全，满足政务和金融监管要求 |
| **成本模式** | 灾备中心按需付费，避免传统自建灾备中心的高投入 |

## 三、云迁移面试Q&A

### Q1：客户要从阿里云迁到天翼云，你怎么做？

**回答框架（5个阶段）：**

#### 第一阶段：调研评估（1-2周）
1. **资源梳理**：使用天翼云RDA工具或手动盘点源端资源
   - 云主机数量、规格（CPU/内存/磁盘）、操作系统版本
   - 数据库类型和版本（MySQL/PostgreSQL/Oracle等）、数据量
   - 对象存储（OSS）数据量
   - 网络架构：VPC规划、子网、安全组规则、负载均衡、CDN
   - 依赖关系梳理：哪些主机/服务之间有调用关系
2. **业务优先级分类**：核心业务 vs 非核心业务，确定迁移顺序
3. **兼容性评估**：OS版本、数据库版本在天翼云是否有对应镜像

#### 第二阶段：方案设计（1周）
1. **网络规划**：在天翼云侧规划VPC、子网、安全组，尽量与源端保持一致
2. **迁移策略选择**：
   - **主机迁移**：使用天翼云CMS（云迁移服务），P2V/V2V方式，整机镜像迁移
   - **数据库迁移**：DTS（数据传输服务）做全量+增量同步
   - **对象存储迁移**：使用迁移工具或rclone等工具做数据搬迁
3. **割接方案**：确定割接时间窗口、回滚方案、验证清单

#### 第三阶段：环境准备（1-2周）
1. 在天翼云创建VPC、子网、安全组、EIP等网络资源
2. 开通对应规格的云主机、RDS、OSS等
3. 打通源端和目标端之间的网络（VPN或专线），用于数据同步
4. 部署迁移工具，做连通性测试

#### 第四阶段：数据迁移与同步
1. **全量迁移**：先做一次全量数据搬迁（主机镜像、数据库全量、存储全量）
2. **增量同步**：持续同步增量数据，保持源端和目标端数据一致
   - 数据库：通过binlog（MySQL）或WAL（PostgreSQL）持续同步
   - 文件存储：增量同步变更文件
3. **一致性校验**：对比源端和目标端数据行数、checksum

#### 第五阶段：割接上线
1. **割接前**：
   - 通知业务方约定停机窗口
   - 确认增量同步追平（无延迟）
   - 在目标端做全面功能测试
2. **割接中**：
   - 源端业务停写（或设为只读）
   - 等待最后一批增量同步完成
   - 切换DNS/负载均衡指向天翼云
   - 验证业务功能和性能
3. **割接后**：
   - 保持源端环境7-14天作为回滚保障
   - 持续监控目标端性能和稳定性
   - 确认无问题后释放源端资源

### Q2：两台不同VPC的云主机要互通怎么办？

**回答：**

有以下几种方案，按场景选择：

**方案一：VPC对等连接（Peering）**
- 适用场景：同一地域内、2-3个VPC互通
- 操作：在VPC控制台创建对等连接，分别在两端VPC的路由表中添加指向对方CIDR的路由
- 优点：同地域免费、延迟低、无带宽限制
- 注意：两个VPC的CIDR不能重叠

**方案二：云企业网（CEN/云联网）**
- 适用场景：3个以上VPC互通、跨地域互通
- 操作：创建云企业网实例，将需要互通的VPC加入，路由自动学习
- 优点：配置简单、支持跨地域、路由自动同步
- 缺点：跨地域需要购买带宽包

**方案三：VPN网关**
- 适用场景：跨云厂商VPC互通
- 操作：两端分别部署VPN网关，建立IPsec VPN隧道
- 优点：加密传输，跨云可用
- 缺点：性能受限于VPN网关带宽

**方案四：高速通道/专线**
- 适用场景：大带宽、低延迟要求
- 操作：通过物理专线打通两端网络

**面试补充要点：**
- 不管哪种方案，都要确认两端VPC网段不冲突
- 互通后还需要配置安全组规则，放行对应端口
- 如果是天翼云，对应产品是"对等连接"和"云联网"

### Q3：云主机无法访问公网，排查思路？

**回答（由外到内，逐层排查）：**

```
第1步：确认是否有公网出口
     ↓
第2步：检查路由表
     ↓
第3步：检查安全组
     ↓
第4步：检查操作系统防火墙
     ↓
第5步：检查DNS
     ↓
第6步：检查应用层
```

**详细步骤：**

1. **确认公网出口**
   - 云主机是否绑定了EIP（弹性公网IP）？
   - 或者所在子网是否配置了NAT网关/SNAT规则？
   - 如果两者都没有，云主机本身就没有公网出口

2. **检查路由表**
   - 查看子网关联的路由表，是否有 `0.0.0.0/0` 的默认路由
   - 下一跳应指向互联网网关（有EIP时）或NAT网关
   - 如果缺少这条路由，流量无法到达公网

3. **检查安全组（出方向）**
   - 确认安全组**出方向**规则是否允许目标端口（如80/443）
   - 默认安全组一般放行全部出方向，但自定义安全组可能限制

4. **检查操作系统防火墙**
   - Linux：`iptables -L -n` 或 `firewalld` 规则
   - Windows：检查Windows防火墙出站规则
   - 临时关闭防火墙测试是否是防火墙导致

5. **检查DNS配置**
   - `ping 8.8.8.8` 能通但 `ping www.baidu.com` 不通 → DNS问题
   - 检查 `/etc/resolv.conf` 中的nameserver配置
   - 尝试使用 `nslookup` 或 `dig` 测试DNS解析

6. **检查网卡和IP配置**
   - `ip addr` 确认网卡UP状态
   - 确认内网IP与云平台分配一致
   - 检查是否有多网卡路由冲突

### Q4：迁移过程中如何保证数据一致性？

**回答：**

数据一致性保障贯穿迁移全过程，分三个层面：

#### 1. 数据库层面
- **全量+增量同步机制**：先做全量迁移，然后通过数据库日志（MySQL的binlog、PostgreSQL的WAL）持续增量同步
- **同步延迟监控**：实时监控增量同步的延迟时间，确保割接前延迟为0
- **一致性校验**：
  - 行数对比：源端和目标端各表行数一致
  - checksum校验：对关键表做数据校验和比对
  - 可使用pt-table-checksum（MySQL）等工具

#### 2. 文件/存储层面
- 使用rsync等工具增量同步，自带校验机制
- 对象存储迁移后做MD5/SHA256校验
- 大文件建议分片校验

#### 3. 割接环节保障
- **停写窗口**：割接时源端设为只读，等最后一批增量同步完成
- **双写验证**（高级方案）：割接前一段时间双写两端，对比结果
- **回滚预案**：保留源端数据不动，割接失败可快速回切

#### 4. 权限隔离
- 数据同步账户和业务账户分开
- 同步账户有增删改查权限
- 业务账户在割接期间设为只读
- 防止同步过程中有人误操作

### Q5：迁移割接窗口怎么定？

**回答：**

割接窗口的确定需要综合考虑以下因素：

#### 1. 业务低峰期
- 分析业务流量曲线，选择访问量最低的时段
- 通常为凌晨2:00-6:00
- 避开月末、季末、年末等业务高峰
- 避开大促、活动期间

#### 2. 预估割接所需时间
- 最后一次增量同步时间（取决于数据变化量）
- DNS切换生效时间（TTL值，建议提前调低至60秒）
- 业务验证时间（功能测试+性能测试）
- 回滚时间预留
- **总窗口 = 同步时间 + 切换时间 + 验证时间 + 回滚缓冲**

#### 3. 割接窗口缩短技巧
- **提前调低DNS TTL**：至少提前24-48小时将TTL调至60秒
- **提前做好增量同步**：确保割接前增量延迟接近于0
- **自动化脚本**：提前编写割接和回滚脚本，减少人工操作时间
- **分批割接**：非核心系统先切，核心系统后切
- **预演练**：正式割接前至少做1-2次全流程演练

#### 4. 割接决策流程
- 制定割接checklist（检查清单）
- 明确Go/No-Go决策点和责任人
- 每个步骤有明确的时间节点和负责人
- 预留充足回滚窗口（建议割接总时间的1.5倍）
- 割接后保留源端环境7-14天

#### 5. 典型割接窗口示例

| 步骤 | 时间 | 动作 |
|------|------|------|
| T-48h | 提前两天 | 调低DNS TTL至60秒 |
| T-2h | 凌晨0:00 | 团队集合、环境确认 |
| T-1h | 凌晨1:00 | 确认增量同步延迟为0 |
| T | 凌晨2:00 | 源端停写/只读 |
| T+15min | 凌晨2:15 | 最后增量同步完成 |
| T+30min | 凌晨2:30 | 数据校验通过 |
| T+45min | 凌晨2:45 | 切换DNS/VIP到天翼云 |
| T+1h | 凌晨3:00 | 业务验证 |
| T+2h | 凌晨4:00 | 确认割接成功/决策回滚 |
