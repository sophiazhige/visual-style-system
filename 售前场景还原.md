# AIGC 数据清洗项目——售前场景还原

> 基于面壁智能、智源研究院实际交付细节，反向还原售前全流程。面试时可作为"我是怎么从0接触客户、设计方案到最终拿下项目"的叙事素材。

---

## 一、客户接触阶段

### 1.1 商机来源

2024年初，大模型赛道进入"军备竞赛"阶段，头部AI公司的核心瓶颈从"模型能不能训"转向"数据够不够好"。金山云销售团队在两个渠道同时捕获到数据清洗的需求信号：

- **面壁智能**：销售团队在跟进面壁的GPU算力采购需求时，发现客户除了训练卡以外，还有大量的CPU算力诉求。深入沟通后得知，面壁正在为MiniCPM系列（1.2B-8B端侧小模型）做数据准备，需要从Common Crawl等公开数据集中清洗出高质量中英文语料。客户之前在自建IDC里跑清洗任务，但算力不足、扩容困难，开始评估上云方案。
- **智源研究院**：智源是金山云的存量KS3（对象存储）客户，此前主要用于存放训练数据。2024年Q1，智源启动了Emu3（具身智能世界模型）项目，需要对260TB+原始视频做大规模清洗。智源内部技术团队主动找到金山云客户经理，询问是否有配套的大数据计算方案。

### 1.2 首次拜访与需求摸底

**面壁首次沟通（约2024年2-3月）：**

我跟随销售一起拜访了面壁的算法负责人和数据团队。这次沟通的目的不是推产品，而是搞清楚三个核心问题：

1. **你们在清洗什么数据？** → 主要是Common Crawl的WARC文件（PB级互联网爬取数据），还有GitHub代码、ArXiv论文、百科等开源数据集。不同来源的格式、编码差异巨大。
2. **你们用什么工具在洗？** → 用的是CCNet框架（Facebook开源），跑MapReduce和Spark任务。清洗流程是：解压缩 → 转换JSON → 抽取txt/html → 语言识别 → 困惑度打分 → 去重。
3. **现在最痛的点是什么？** → 面壁内部多个团队在同时跑不同的数据配比实验（"模型风洞"方法论），共用一套集群，一个团队的MapReduce任务把CPU拉满，其他团队就被卡住，直接影响MiniCPM迭代速度。他们需要的不只是算力，而是**多团队并行不干扰**。

**智源首次沟通（约2024年3月）：**

智源的诉求更直接——他们已经明确知道自己要做什么（清洗260TB视频数据），核心关注的是：

1. **成本**：视频数据清洗淘汰率极高（预计97%会被丢弃），如果用传统方案，97%的本地存储投入是浪费的。
2. **吞吐**：Emu3有训练deadline，清洗不能拖太久，需要高并发下载和处理。
3. **稳定性**：不能清洗到一半任务挂掉从头再来。

## 二、业务场景调研与技术方案设计

### 2.1 调研方法论

拿到客户的初步需求后，我没有急着出方案，而是做了三件事：

**第一，拆解客户的业务流程，理解数据流向。**

我让客户的数据工程师画了一遍他们现有的数据处理pipeline：
- 文本清洗流程：原始WARC → 解压缩/转JSON → 抽取正文 → 去重（MinHash） → 语言识别（fastText） → 困惑度打分（KenLM） → 质量过滤
- 视频清洗流程：原始视频 → 场景分割 → 抽帧 → OCR/美学评分（GPU） → caption生成（GPU） → 元数据聚合 → 过滤

**第二，量化关键技术指标。**

- 面壁：数据总量8PB+，每轮清洗淘汰率~90%，一季度跑5-8轮配方实验，每轮需要全量重跑去重和过滤流程。文本是海量小文件（几十万到百万级），随机读写密集。
- 智源：数据总量260TB视频，淘汰率~97%，最终只保留几十GB元数据。视频是大文件，顺序读写为主。

**第三，评估客户当前方案的瓶颈。**

我拿到了客户在自有环境跑的性能数据，发现：
- 面壁的瓶颈在I/O：Spark任务处理海量小文件时，产生大量Shuffle Data（临时中间文件），如果走网络读存算分离的远端存储，几十万并发会把网关打满。
- 智源的瓶颈在成本：如果用存算一体，每台机器都要配12块16T本地盘，但97%的数据处理完就删了，本地存储投入大部分浪费。

### 2.2 方案设计：差异化架构

基于调研结论，我给两个客户设计了截然不同的架构方案：

**面壁 → 存算一体 + 资源强隔离**

- 架构选择逻辑：文本清洗是CPU密集 + I/O密集的混合任务，海量小文件的随机读写对底层IOPS和元数据性能要求极高。存算分离方案下，频繁的网络读取会产生严重带宽瓶颈。
- 方案要点：万核裸金属集群（8358P高性能机型），本地挂载大盘（单机12×16T），RAID0并发读写。通过KMR的YARN队列做严格资源池隔离，每个团队有独立的计算资源配额。原始数据和结果数据留KS3作为保底。
- 卖点：5000核/120小时完成500TB清洗，CPU负载率200%，各团队并行互不干扰。

**智源 → 存算分离 + 弹性KS3**

- 架构选择逻辑：视频是大文件、顺序读写，97%淘汰率意味着存算一体方案的存储成本几乎全部浪费。
- 方案要点：数据统一存在KS3对象存储，70+台裸金属按需拉取处理，计算与存储独立扩缩容。下载带宽压到150Gbps。
- 卖点：存储成本只和最终有效数据挂钩，而不是和原始数据总量挂钩。

### 2.3 TCO测算与成本对比

这是打动客户的关键环节。以智源为例，我做了一份两套方案的TCO对比表：

| 维度 | 存算一体方案 | 存算分离方案 |
|------|------------|------------|
| 计算资源 | 70台裸金属 (8358P) | 70台裸金属 (8358P) |
| 存储资源 | 每台12×16T本地盘 = 13.4PB | KS3按量付费，260TB存储 |
| 存储浪费 | 清洗后97%存储闲置 | 淘汰数据直接删除，不占存储成本 |
| 扩缩容 | 计算和存储绑定，无法独立扩缩 | 计算和存储独立弹性 |
| 总成本差异 | 基准 | 较存算一体节省约40-50%存储成本 |

对面壁，TCO测算的核心不是存储成本，而是**时间成本**——如果多团队抢资源导致清洗实验延迟1周，对于和时间赛跑的小模型公司来说，竞争力损失远大于多花的存储费用。

## 三、POC设计与验证

### 3.1 POC的定位

POC不是"免费试用"，而是**用最小资源验证方案可行性，同时暴露规模化部署的风险**。

### 3.2 面壁POC设计

**POC目标：**
1. 验证存算一体方案下，MapReduce和Spark任务的I/O性能是否满足海量小文件场景
2. 验证多租户资源隔离是否有效，两个团队同时跑任务互不影响
3. 建立性能基线（Benchmark），为后续报价和扩容提供数据支撑

**POC资源配置：**
- 计算：先给2000核做初步测试（约20台裸金属）
- 存储：本地盘 + KS3双存储
- 测试数据：客户提供50TB Common Crawl子集
- 测试周期：2周

**POC验收标准：**
- 性能指标：MapReduce任务零错误完成50TB清洗，CPU负载率≥180%
- 隔离指标：两个队列同时跑满，互不影响对方任务完成时间
- 稳定性指标：连续72小时无任务超时或OOM

**POC过程中发现的关键问题：**

POC阶段实际上暴露了一个重要问题——客户最初用的是存算分离集群（走网络读KS3），MapReduce长任务一直跑不通，Spark任务效率也很慢。经过驻场排查：
- Spark任务超时：桶限速200Gb，对Spark大量并发请求冲击很大
- Socket reset：KS3流量触发限流导致任务失败
- 裸金属单节点带宽超限：单节点15Gb触发PGW限流丢包

这个排查结果反过来验证了**存算一体才是正确架构选择**——最终将存算分离节点全部改配加盘，加入存算一体集群后，500TB MapReduce零错误完成。

### 3.3 智源POC设计

**POC目标：**
1. 验证存算分离架构下，KS3在高并发下载场景的吞吐稳定性
2. 验证视频清洗pipeline（场景分割→抽帧→OCR→caption）的端到端性能
3. 测算不同并发度下的带宽需求，为正式扩容做预判

**POC资源配置：**
- 计算：先用10-15台裸金属
- 存储：KS3对象存储
- 带宽：初始50Gbps，根据测试结果逐步调整
- 测试数据：客户提供30TB视频子集
- 测试周期：1-2周

**POC验收标准：**
- 吞吐指标：并行下载带宽稳定在目标值，无限流丢包
- 清洗指标：30TB视频在目标时间内完成全流程清洗
- 稳定性指标：连续运行无任务中断

## 四、报价策略

### 4.1 定价逻辑

金山云的数据清洗方案报价通常包含以下几个组成部分：

| 计费项 | 说明 |
|--------|------|
| 裸金属租赁费 | 按台/月计费，根据机型（如8358P）定价 |
| KS3对象存储费 | 按存储容量（TB/月）+ API调用次数计费 |
| KMR集群管理费 | 按管理节点数量计费（Spark/MR调度平台） |
| 网络带宽费 | 专线/公网带宽费用，按Gbps计费 |
| 数据传输费 | 跨Region或跨境数据传输按量计费 |

### 4.2 面壁报价要点

- **核心卖点：性能和多租户隔离**，客户对价格敏感度相对低（时间成本更重要）
- 初始报价基于5000核规模，包月模式
- 后续随着客户清洗任务的峰谷波动，我主动推荐了Serverless方案，给客户算了一笔账：Serverless比EPC（弹性物理机）便宜约2/3，帮助客户大幅降低了闲时成本
- 商务策略：初期用包月裸金属打标杆案例，中后期用Serverless Spark做增量收入

### 4.3 智源报价要点

- **核心卖点：存储成本优势**，TCO对比是决定客户选我们的关键
- 裸金属按需使用 + KS3按量付费，客户只为有效数据买单
- 带宽成本：150Gbps的下载带宽是大头，通过预购带宽包降低单价
- 商务策略：先做标杆案例，沉淀Benchmark数据，后续用于其他AI客户的售前打单

### 4.4 Serverless产品的定价突破

在面壁项目后期，我发现了客户"脉冲式算力需求"的痛点（清洗实验期间CPU拉满，间歇期集群闲置），于是主动推动Serverless Spark的POC：

- 资源申请：向内部申请3000CU的Serverless免费公测资源
- 客户激励：免费公测一个月、金山云驻场保障、不设SLA
- 报价对比：给客户做了Serverless vs EPC vs 友商的单价对比表，Serverless单价低于EPC 2/3，也低于友商同类产品

这个策略本质上是**用POC资源成本换三方共赢**——客户获得弹性算力降成本、公司获得标杆客户、产品获得真实场景压测。

## 五、竞品分析（2024年初数据清洗市场）

### 5.1 竞争格局

2024年初，能做大规模AIGC数据清洗的云厂商主要有以下几家：

| 厂商 | 核心产品 | 优势 | 劣势 |
|------|---------|------|------|
| **阿里云** | EMR（Elastic MapReduce）+ OSS + MaxCompute | 产品成熟度最高，EMR支持Spark/MR/Flink全栈；OSS存储规模大、稳定性强；有Serverless EMR | 价格偏高；大客户定制灵活度不如小厂；对AI场景的针对性方案沉淀少 |
| **火山引擎（字节跳动）** | BMR（云原生大数据平台）+ TOS对象存储 | 字节内部有海量数据处理经验背书；TOS性能好；有一定的Serverless能力 | 外部商业化时间短，企业客户案例积累不足；2024年初还在拓客阶段 |
| **华为云** | MRS（MapReduce Service）+ OBS | 政企客户基础强；国产化替代优势；昇腾生态可打CPU+GPU混合清洗 | 对互联网/AI初创客户吸引力弱；MRS产品功能更新节奏慢于阿里EMR |
| **AWS** | EMR + S3 | 全球最成熟的大数据生态；S3稳定性和性能标杆；Spot Instance可大幅降低清洗成本 | 国内合规限制（数据出境）；网络延迟；无本土支持团队 |
| **百度云** | BMR + BOS | 百度内部搜索数据清洗经验；有文心大模型生态协同优势 | 大数据产品商业化规模小；AI客户偏好自建或选阿里/AWS |

### 5.2 金山云的差异化打法

在面对上述竞争对手时，金山云的数据清洗方案不是靠产品功能取胜（EMR产品成熟度确实不如阿里），而是靠以下差异化：

**1. 贴身服务 + 方案定制能力**

大模型客户的数据清洗需求高度非标。面壁需要多租户隔离、智源需要150Gbps的极限吞吐——这些在标准产品里找不到现成方案。金山云的优势是小厂的灵活性：架构师可以驻场、方案可以定制、资源可以快速调配。大厂的标准化流程反而成了劣势。

**2. 裸金属 + 大数据平台的组合能力**

金山云的KMR直接纳管裸金属资源，不走虚拟化层，计算性能接近物理机极限。对于数据清洗这种CPU密集场景，裸金属的性能优势明显（对比阿里EMR走ECS虚拟机，有5-10%的性能损耗）。

**3. 存储性价比**

KS3对象存储在AI场景的定价策略相对激进（尤其是大容量长期存储），加上全闪存储桶对吞吐敏感型场景的支持，在智源这类大规模视频清洗场景中有成本优势。

**4. Serverless先发优势（2024年Q3）**

金山云Serverless Spark在2024年9月上线公测时，在AI数据清洗场景的竞品中属于较早落地的。阶跃星辰的POC测试中，金山云1500核Serverless Spark跑数据清洗+中模型推理任务耗时2.4小时，友商用1000核（CPU代次高2代）需要4.5小时。这个Benchmark成为后续打单的核心武器。

### 5.3 与友商的实际交锋

**阶跃星辰项目中的直接对标：**

阶跃星辰在评估数据清洗方案时同时测了金山云和某友商（大概率是阿里或火山）。对比结果：

| 维度 | 金山云 Serverless Spark | 友商 |
|------|----------------------|------|
| 资源 | 1500核 | 1000核 |
| CPU代次 | 当前代 | 高2代 |
| 清洗+推理耗时 | 2.4小时 | 4.5小时 |
| 性能结论 | 在CPU代次更低的情况下，靠Serverless调度优化和存储吞吐取胜 | 硬件更强但整体效率不如金山云 |

## 六、完整售前叙事（面试版本）

> 以下是面试时可以用的完整故事线，以面壁项目为主线：

"这个项目的商机来源是我们销售在跟面壁谈GPU算力采购时，发现他们除了训练卡，还有大量CPU算力诉求。我跟着销售去拜访了面壁的算法和数据团队，做了一次深度的业务调研。

**调研阶段**，我重点搞清楚了三件事：他们用CCNet框架在洗Common Crawl数据，文本清洗的核心pipeline是解压缩→JSON转换→去重→语言识别→困惑度打分；他们的痛点不是单纯的算力不够，而是多团队并行实验互相抢资源，直接影响MiniCPM的迭代速度；以及文本数据的特点——海量小文件、随机读写、高并发，这决定了架构选型。

**方案设计阶段**，我基于调研结论判断存算一体才是正解，因为海量小文件走网络读远端存储，几十万并发会把网关打满。我设计了裸金属+本地大盘+KMR资源隔离的方案，同时做了TCO测算——存算一体虽然存储成本高于存算分离，但多团队并行效率带来的时间收益远超存储差价。

**POC阶段**，我们先给了2000核做测试。POC过程中实际上还验证了一个关键假设——客户最初尝试过存算分离方案，MapReduce长任务一直跑不通，我驻场排查发现是桶限速和单节点带宽超限导致的。这个排查结果反过来加固了客户对存算一体方案的信心。最终POC验收指标全部达标：5000核/120小时/500TB/零错误。

**报价阶段**，初期用包月裸金属锁定客户。但后续我发现客户清洗任务有明显的峰谷波动，于是主动推荐了Serverless方案，给客户算了一笔账——Serverless比传统EPC便宜2/3。我向内部申请了3000核免费公测资源，和客户谈了一个月免费测试+驻场保障的方案，最终推动了Serverless Spark的首个商业化客户落地。

**竞争层面**，我们面对的主要是阿里EMR和火山引擎的竞争。金山云不是靠产品功能取胜——EMR的成熟度确实不如阿里——而是靠三点：一是小厂灵活性，方案可定制、架构师可驻场；二是裸金属的极致性能，不走虚拟化层；三是Serverless的性价比，在阶跃星辰的对标测试中，我们1500核跑出了友商1000核（CPU高2代）的近两倍效率。"

## 七、售前关键能力清单（用于面试自我总结）

| 能力项 | 在本项目中的体现 |
|--------|----------------|
| 需求挖掘 | 从GPU算力采购需求中发现数据清洗的衍生需求 |
| 业务理解 | 深入理解CCNet框架、清洗pipeline、数据特性（小文件vs大文件） |
