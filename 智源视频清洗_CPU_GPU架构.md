# 智源视频清洗 CPU+GPU 同机协同架构

> 项目背景：智源Emu3（具身智能世界模型），260TB原始视频，清洗淘汰率97%，存算分离架构。

---

## 一、核心设计思路：同一台机器内 CPU→GPU 流水线

不是CPU集群和GPU集群两个独立的集群，而是**每台裸金属上CPU和GPU协同工作**——CPU做预处理（视频解码/切分/抽帧），产出的中间数据通过本机内存直接喂给本机GPU做推理（OCR/美学评分/Caption），避免跨节点的数据搬运开销。

70+台裸金属组成一个统一集群，由Spark/Ray统一调度，每台机器就是一条完整的处理流水线。

## 三、单节点内部架构详解

### 3.1 硬件配置（单台裸金属）

| 组件 | 配置 | 作用 |
|------|------|------|
| CPU | Intel Xeon 8358P × 2（64核） | 视频解码、场景切分、抽帧 |
| GPU | A100/A800 80GB × 4-8卡 | OCR、美学评分、视觉推理、Caption生成 |
| 内存 | 1TB DDR4 | CPU→GPU数据中转缓冲区 |
| 本地SSD | 1-2块 NVMe（~2TB） | 临时缓存，视频解码中间产物 |
| 数据网 | 25GbE × 2 | 连KS3拉源数据 + 写回结果 |

> 注意：这里不配12×16T本地大盘（那是存算一体的面壁方案）。存算分离架构下，本地只配少量SSD做临时缓存，源数据和结果数据都在KS3上。

### 3.2 单节点内部数据流

```
┌─────────────────── 一台裸金属 ───────────────────────────┐
│                                                          │
│  ① KS3 ──25GbE──→ CPU：拉取视频到内存                      │
│                     │                                    │
│  ② CPU：视频解码 → 场景切分 → 抽帧                          │
│         │                    │                           │
│         │  视频分片           │  关键帧图片                 │
│         │  (内存/本地SSD)      │  (内存)                   │
│         │                    │                           │
│         ▼                    ▼                           │
│  ③ GPU：                                                 │
│     ┌─────────────────┐  ┌─────────────────┐            │
│     │   视频分片        │  │   关键帧图片     │            │
│     │       ↓          │  │       ↓         │            │
│     │  视觉模型推理     │  │   OCR文字识别    │            │

│              ▼                     ▼                     │
│  ④ 聚合结果：                                             │
│     - 视频ID + caption文本 + 质量分数 + 保留/淘汰标记        │
│     - 保留的视频分片（3%）                                  │

关键：步骤②和③在同一台机器内通过内存传递，不走网络。
CPU不断产出下一批数据，GPU不断消费上一批数据，形成机内流水线。
```

### 3.3 CPU和GPU如何在同一台机器上流水线并行？

CPU和GPU不是"先后执行"，而是**重叠运行**：

```
时间轴 →
CPU:  [拉取+解码 batch1] [拉取+解码 batch2] [拉取+解码 batch3] ...
GPU:                     [推理 batch1]       [推理 batch2]      [推理 batch3] ...
                         ↑
                  CPU产出batch1后，GPU立刻开始处理
                  同时CPU已经在准备batch2了
```

- CPU解码一个视频片段可能只要几秒（CPU密集但计算量相对小）
- GPU做一次Caption推理可能要几十秒（模型大、自回归生成慢）
- 所以GPU是瓶颈端，CPU总是能提前准备好数据等GPU来取
- 每台机器上的Spark/Ray Actor负责编排这个生产者-消费者关系

### 3.4 为什么要在同一台机器上跑，而不是分成独立的CPU集群和GPU集群？

| 维度 | 同机协同（实际方案） | 分离集群 |
|------|-------------------|---------|
| 数据搬运 | CPU产出直接通过内存给GPU，**零网络开销** | 中间数据要先写KPFS/KS3，GPU再从网络读回来，额外消耗带宽 |
| 网络压力 | 网络只承担两个方向的流量：①KS3→节点（拉源数据）②节点→KS3（写结果） | 多出一个方向：CPU集群→中转存储→GPU集群，带宽可能翻倍 |
| 延迟 | 内存传递延迟ns级 | 跨节点网络传输延迟ms级 |
| 架构复杂度 | 单集群统一调度 | 两个集群 + 中转存储层（KPFS），运维复杂度高 |
| 资源利用 | GPU等数据时，本机CPU一定能及时供上（CPU处理速度 >> GPU推理速度） | CPU集群可能已经跑完了，GPU集群还在慢慢消费，CPU资源提前释放但也提前浪费 |

**一句话总结**：视频清洗的中间数据（抽帧图片、视频分片）体量大、GPU消费频率高，走网络传输既浪费带宽又增加延迟，放在同一台机器内通过内存传递是最自然的选择。

## 四、集群层面的数据流与调度

### 4.1 整体数据流（70+台并行）

```
KS3(260TB原始视频)
    │
    │ 70+台裸金属并行拉取，聚合带宽150Gbps
    │ 每台 ~2Gbps，每台同时处理约 260TB/70 ≈ 3.7TB 数据
    │
    ├──→ 节点1: CPU解码切分抽帧 → 内存 → GPU推理打分 → 结果写回KS3
    ├──→ 节点2: CPU解码切分抽帧 → 内存 → GPU推理打分 → 结果写回KS3
    ├──→ 节点3: CPU解码切分抽帧 → 内存 → GPU推理打分 → 结果写回KS3
    │    ...
    └──→ 节点70: CPU解码切分抽帧 → 内存 → GPU推理打分 → 结果写回KS3
    
    │
    ▼
KS3(结果数据)
    ├── meta文件：几十GB（视频ID、质量分、caption、保留/淘汰标记）
    ├── 保留的视频分片：~8TB（260TB × 3%）
    └── 淘汰日志

总耗时：~34小时
瓶颈端：GPU推理（Caption生成的自回归推理最耗时）
```

### 4.2 调度框架

基于**Spark**编排（你的PDF里明确写了视频清洗基于Spark框架）：

- KMR（金山云托管大数据平台）纳管70+台裸金属，统一调度CPU和GPU资源
- 每台机器上运行多个Spark Executor / Ray Actor
- 每个Actor内部串联 CPU预处理→GPU推理 的pipeline
- Actor之间无数据依赖，完全并行（embarrassingly parallel）
- Spark负责任务分发（哪个Actor处理哪些视频文件）和容错重试

### 4.3 资源配比的设计逻辑

70+台机器的CPU和GPU配比是经过benchmark调过的：

- CPU侧：64核跑视频解码和切分，吞吐远高于GPU推理速度 → CPU大部分时间在等GPU → CPU不是瓶颈
- GPU侧：4-8张A100跑视觉推理和Caption → 单张A100做7B模型推理大约每秒处理1-2个视频片段 → GPU是瓶颈
- 所以GPU卡数决定了每台机器的处理速度，CPU只要"够用"就行

如果发现GPU吞吐不够，不需要加CPU——只需要加更多的GPU节点。这也是存算分离架构的好处：源数据在KS3上，随时可以加新节点来分摊任务，不用搬数据。

## 五、GPU上跑的两条推理线详解

### 5.1 图片处理线（处理抽帧产出的关键帧）

```
关键帧图片（从内存接收）
    │
    ▼
OCR文字识别
    ├─ 模型：PaddleOCR / EasyOCR（轻量，百M级参数）
    ├─ 作用：识别图片中的文字
    ├─ 清洗逻辑：
    │   - 大面积水印 → 标记"劣质"
    │   - 广告文字 / 平台logo → 标记"劣质"
    │   - 正常文字 → 提取为文本标签
    ├─ 性能：单卡A100可达数百张/秒（轻量模型）
    │
    ▼
美学评分（Aesthetics Score）
    ├─ 模型：CLIP-based（如LAION Aesthetics Predictor，百M级参数）
    ├─ 作用：给图片打0-10质量分
    ├─ 清洗逻辑：
    │   - 分数 < 阈值（如4.5）→ 淘汰（模糊、过曝、构图差）
    │   - 分数 ≥ 阈值 → 保留，分数作为质量权重
    ├─ 性能：单卡A100可达数百张/秒
    │
    ▼
输出：每张图片的标签（OCR文字 + 美学分数 + 保留/淘汰）
```

### 5.2 视频处理线（处理切分产出的视频分片）

```
视频分片（从内存/本地SSD接收）
    │
    ▼
视觉模型推理
    ├─ 模型：多模态视觉模型（InternVL / QwenVL等，7B-14B参数）
    ├─ 输入：视频分片按间隔采样成若干帧，拼接输入模型
    ├─ 作用：模型"观看"视频，理解场景内容
    │
    ▼
Caption生成（自回归文本生成）
    ├─ 输出："一只机械臂从桌面拿起红色方块，放入右侧容器中"
    ├─ 清洗逻辑：
    │   - 描述与具身智能无关（纯风景、人脸特写等）→ 淘汰
    │   - 包含物体交互、空间运动、操作动作 → 保留
    ├─ 性能：单卡A100约每秒1-2个片段（模型大 + 自回归慢）
    │
    ▼
输出：每个视频分片的结构化描述 + 保留/淘汰标记
```

### 5.3 同一台机器上GPU卡的分配

每台机器有4-8张A100，按任务类型分组：

```
假设单台 8张 A100：

┌───────────────────────────────────────┐
│  图片线：2张卡                          │
│  - OCR + 美学评分都是轻量模型            │
│  - 2张卡的吞吐已经远高于CPU抽帧的产出速度 │
├───────────────────────────────────────┤
│  视频线：6张卡                          │
│  - 多模态视觉模型7B-14B，单次推理耗时长   │
│  - Caption自回归生成，吞吐天然低          │
│  - 需要更多卡来匹配CPU的视频切分产出速度   │
│  - 大模型可能需要2卡tensor parallel      │
└───────────────────────────────────────┘
```

比例约 **1:3（图片线:视频线）**，因为视频Caption是整个pipeline里计算量最大、吞吐最低的环节。

## 六、关键设计决策（面试追问准备）

### Q1: 为什么选存算分离而不是存算一体？

> "视频是大文件、顺序读写，KS3的吞吐完全能支撑。核心是成本——97%的数据最终被淘汰，如果存算一体，每台机器配12×16T本地盘，这些存储97%是在为垃圾数据买单。存算分离让客户只为有效数据付存储费。"

### Q2: 每台机器上CPU和GPU的速度不匹配怎么办？

> "CPU解码切分很快（秒级），GPU推理很慢（十秒级），所以GPU永远是瓶颈。但这不是问题——CPU跑完一批就写入本机的内存队列或本地SSD缓存，继续处理下一批。GPU按自己的速度从缓存里取，不会出现GPU等CPU的情况。本机内存作为缓冲区，天然解耦了两者的速度差异。如果缓冲区满了，CPU就暂停拉取，等GPU消费一部分后再继续——这就是标准的生产者-消费者模型。"

### Q3: 为什么不把CPU预处理单独放几台便宜的纯CPU机器上，GPU推理放GPU机器上？

> "因为中间数据的搬运成本太高。视频抽帧后会产生大量的图片文件，如果走网络从CPU机器传到GPU机器，等于额外消耗一倍的网络带宽。我们150Gbps的带宽已经全给了KS3拉源数据，没有余量再去搬运中间数据了。放在同一台机器内通过内存传递，零网络开销，是最高效的选择。而且GPU机器本身就有CPU——64核CPU做视频解码绰绰有余，没必要单独再搞一批纯CPU机器。"

### Q4: 70+台机器怎么分任务？

> "Spark/Ray做任务分发，把260TB视频按文件粒度拆成数万个task，均匀分配给70+台节点。每个节点处理分配到的那批视频文件，节点之间没有数据依赖，完全并行（embarrassingly parallel）。如果某个节点处理慢了或挂了，Spark会自动把它的任务重新分配给其他节点。"

### Q5: 34小时是怎么来的？

> "每台机器处理约3.7TB数据（260TB/70台）。GPU是瓶颈端，以8张A100的吞吐来算，单台处理3.7TB视频大约需要30-34小时。70台并行同时跑，所以总耗时就是单台耗时——约34小时。如果要缩短时间，加更多的GPU节点就行，存算分离架构下加节点很方便，源数据在KS3上，新节点直接拉取分配的文件即可。"

## 七、面试讲述版本（2分钟）

> "智源的视频清洗架构，核心是70+台GPU裸金属同机协同的并行处理。每台机器同时配了64核CPU和4-8张A100——CPU负责从KS3拉取视频、做解码切分和抽帧，产出的图片和视频分片直接通过本机内存传给本机GPU，GPU跑两条推理线：图片线做OCR和美学评分过滤低质帧，视频线用多模态视觉模型看视频片段生成Caption。
>
> 之所以CPU和GPU放同一台机器而不是分开，核心原因是**避免中间数据的网络搬运**。视频抽帧会产出大量图片，如果走网络传到另一个集群的GPU上，会额外消耗150Gbps的带宽——而这个带宽我们已经全部给了从KS3拉源数据。放在同一台机器内，内存传递零网络开销。
>
> 70+台机器完全并行、互不依赖，Spark按文件粒度分发任务。GPU推理是整条链路的瓶颈——Caption生成用的是7B级别的多模态模型，自回归推理吞吐低。所以总耗时34小时，本质上是被GPU推理速度决定的。
>
> 存算分离架构下，如果客户要缩短清洗时间，直接加GPU节点就行，源数据在KS3上，新节点随时可以加入分担任务。同时97%的淘汰数据不占本地存储成本——这是和存算一体方案最大的成本差异。"
